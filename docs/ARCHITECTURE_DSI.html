1Synthèse exécutive

Gestion Marchés Unicancer est une application web métier dédiée au suivi des marchés publics du groupe Unicancer (19 Centres de Lutte Contre le Cancer). Elle couvre le cycle complet : sourcing → ouverture → analyse → notation → attribution → reporting.

Application React SPA sans serveur applicatif, sans base de données distante. Toutes les données sont stockées localement dans le navigateur (localStorage) et alimentées par des fichiers Excel/PDF/Word importés par l'utilisateur. L'application est déployable sur n'importe quel hébergement statique (GitHub Pages, Azure Static Web Apps, etc.).

Cas d'usage couverts

Gestion documentaire des AO

Import et analyse automatisée des offres fournisseurs (PDF/DOCX/XLSX) : extraction de contacts, standardisation des BPU, suivi des pièces, contrôle qualité.

Notation des candidats

Tableau collaboratif d'évaluation par critères pondérés, import Excel, export PDF, historique des sessions de notation.

Reporting d'exécution

Suivi CA, tonnages, maintenances et renouvellements ; construction automatique de la BDD investissements à partir des reportings fournisseurs.

Annuaire CLCC & formations

Référentiel des 491 contacts CLCC (19 centres × 27 fonctions), gestion des formations, sync optionnelle Outlook.

3Architecture globale

L'application suit une architecture monolithique frontend : un seul bundle JavaScript contenant l'ensemble de la logique métier. Les données proviennent de trois sources : des données de référence en dur, des fichiers importés par l'utilisateur, et le localStorage du navigateur pour la persistance.

Vue d'ensemble : application React 100% client-side avec persistance localStorage et intégrations optionnelles

Principes architecturaux

📦 Organisation par feature

Le code est groupé par domaine métier (features/ao, features/notation, features/reporting…) plutôt que par couche technique. Chaque module est largement autonome.

🔄 État via Context API

Pas de Redux ni de Zustand : l'état global est réparti dans 9 React Contexts spécialisés, chacun avec sa propre logique de persistance localStorage.

📂 Traitement fichiers client

Toute l'analyse de fichiers (PDF, DOCX, XLSX) se fait dans le navigateur : aucun fichier n'est envoyé sur un serveur, garantissant confidentialité et conformité.

🚀 Déploiement statique

Un vite build produit un dossier dist/ servable par n'importe quel CDN ou hébergement statique. Aucun runtime serveur n'est requis en production.

Logique de redirection intelligente

Sur /marche/:id, l'application choisit dynamiquement la page la plus pertinente :

function MarcheRedirect() {
  const { id } = useParams();
  const { getSession } = useNotation();
  const m = marches.find(x => x.id === id);

  if (!m) return <Navigate to="/" replace />;

  // 1) Session de notation active → onglet Notation
  if (getSession(id)) return <Navigate to={`/marche/${id}/notation`} replace />;

  // 2) Données reporting disponibles → onglet Reporting
  if (APP_ID_TO_MARCHE[id]?.length) return <Navigate to={`/marche/${id}/reporting`} replace />;

  // 3) Sinon, défaut Notation
  return <Navigate to={`/marche/${id}/notation`} replace />;
}
🔓 Authentification
Toutes les routes sont actuellement accessibles sans authentification. La configuration MSAL/Azure AD est présente dans le code (utils/msalConfig.js) mais non activée — elle nécessite la définition des variables d'environnement VITE_AZURE_CLIENT_ID et VITE_AZURE_AUTHORITY pour être opérationnelle.

7État applicatif — les 9 Contexts

L'état global de l'application est réparti dans 9 React Contexts spécialisés. Chacun encapsule sa logique de lecture/écriture et sa persistance localStorage. Cette approche évite Redux/Zustand au prix d'une imbrication de Providers dans App.jsx.

Imbrication des Providers dans App.jsx (9 niveaux)

Détail des Contexts

ContextRôlePersistance
NotationContext Sessions de notation par marché + historique (100 sessions max) gm-notation-history
MarcheMetaContext Métadonnées éditables (responsable, tags, dates…) overlay sur mockData gm-metas
NewMarchesContext Drafts de nouveaux marchés en création localStorage
NewFormationsContext Drafts de nouvelles formations localStorage
FormationsMetaContext Métadonnées éditables des formations localStorage
ReportingDataContext Dataset Excel (Suivi_Invest.xlsx) parsé en mémoire gm-reporting-data
BddPendingContext Queue de lignes BDD en attente d'export, indexées par marché gm-bdd-pending
SourcingTemplatesContext Templates sourcing (critères, pondérations) localStorage
ShortcutsContext Registre des raccourcis clavier actifs Mémoire seule
⚠️ Limites du localStorage
Le quota standard du localStorage est de 5 à 10 Mo par domaine. Les Contexts ReportingDataContext et NotationContext stockent potentiellement de gros volumes (Excel parsés, historique). Un dépassement provoquerait une exception silencieuse. En cas de migration vers un usage multi-utilisateur, une base distante deviendra nécessaire.

10Authentification & Copilot

10.1 — Authentification Azure AD (préparée, non activée)

La librairie @azure/msal-browser est configurée dans src/utils/msalConfig.js en mode SPA avec stockage des tokens dans le localStorage. Scopes demandés : User.Read et Contacts.ReadWrite (pour Outlook sync).

Configuration requise pour activer l'auth
Définir les variables d'environnement Vite suivantes dans un fichier .env : 
VITE_AZURE_CLIENT_ID=<app-id Azure AD>
VITE_AZURE_AUTHORITY=https://login.microsoftonline.com/<tenant-id>
Sans ces variables, l'authentification reste inactive et l'application est ouverte.

Cas d'usage de MSAL dans l'application

  • outlookSync.js — création d'événements calendrier (échéances marché) et de contacts Outlook depuis l'annuaire CLCC
  • Préparé pour un futur SSO Unicancer (non activé)

10.2 — Copilot (assistant IA)

Un widget chatbot (CopilotWidget.jsx) accessible depuis toutes les pages interroge un modèle Claude (Anthropic) via un proxy Node local.

Flux Copilot — la clé API Anthropic n'est jamais exposée au client
Sécurité du proxy Copilot
La clé ANTHROPIC_API_KEY est stockée côté serveur (variable d'environnement Node) et n'est jamais exposée au navigateur. Le proxy copilot-proxy.mjs doit être lancé séparément (npm run copilot) et tourne uniquement en local sur le port 3001. Pour un déploiement, il faudrait héberger ce proxy (Azure Functions, Cloud Run, etc.) ou désactiver le widget.

12Synthèse DSI

Forces de l'architecture

✅ Souveraineté des données
Aucune donnée n'est envoyée vers un serveur externe (hors Copilot optionnel). Le traitement des fichiers offres se fait entièrement dans le navigateur.
✅ Déploiement simple
Bundle statique servable par n'importe quel hébergement. Pas de stack serveur à maintenir, pas de base de données à administrer.
✅ Stack moderne et stable
React 18 + Vite 5 = écosystème mainstream, communauté large, montée de version sereine.
✅ Code organisé par domaine
La séparation par feature facilite l'évolution indépendante des modules et l'onboarding de nouveaux contributeurs.

Points de vigilance

⚠️ Pas d'authentification active
L'application est aujourd'hui accessible sans contrôle d'accès. À activer (MSAL) avant tout déploiement hors environnement de test.
⚠️ Données mono-utilisateur
Le localStorage est par-navigateur, par-utilisateur. Aucun partage natif des notations/métadonnées entre membres de l'équipe.
⚠️ Quota localStorage (5-10 Mo)
Risque de saturation à terme avec l'accumulation de sessions de notation et de gros fichiers Excel parsés.
⚠️ Données de référence en dur
La liste des marchés (mockData.js) et l'annuaire CLCC sont versionnés dans le code. Toute mise à jour requiert un nouveau déploiement.

En résumé

Profil applicatif
Outil interne d'aide à la décision et au pilotage des marchés, parfaitement adapté à un usage individuel ou en petite équipe (≤ 10 utilisateurs) où chacun travaille sur son périmètre avec des données partagées par export/import de fichiers. Pour passer à un usage collaboratif large, l'ajout d'un backend devient un préalable.

Le projet est techniquement sain : stack moderne, séparation claire des responsabilités, code lisible, pipelines de traitement de fichiers solides. Sa principale limite est structurelle (architecture client-only) et non liée à la qualité du code. Les évolutions vers un mode multi-utilisateur centralisé sont possibles sans réécriture majeure grâce à l'isolation des couches de données via les Contexts.
