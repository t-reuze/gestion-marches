2. Les 4 sources d'information

L'algorithme combine quatre sources pour remplir chaque case BDD :

① Reporting fournisseur

Pour chaque BC : Lot, Date d'achat, Établissement (nom + ville), Désignation, Quantité, Prix TTC, etc.

② Allotissement

Liste officielle des lots du marché avec leur intitulé. Utilisé pour confirmer que les n° de lot des reportings sont valides.

③ Onglet « Nomenclature CLCC et EA » du Suivi_Invest

93 entrées de référence (CLCC + Établissements affiliés) avec leur code canonique. C'est la table de vérité pour la colonne B CLCC unique.

④ Configuration marché (marcheInvestConfig.js + apprise depuis la BDD)

Pour chaque marché : libellé Excel attendu, format de lot, mapping lot → type d'équipement, range consommables, taux Gain/Achats par défaut. Voir §7.

4. Le matching CLCC/EA en détail

Pour chaque ligne d'un reporting, on doit transformer un libellé brut comme "Centre Eugène Marquis - Rennes" ou "GH HEGP-BROUSSAIS" en un code canonique (Rennes, APHP_Pompidou) qui sera écrit en colonne B de la BDD.

L'algorithme tente 4 stratégies dans l'ordre :

① Alias utilisateur appris (priorité absolue)

Si l'utilisateur a déjà corrigé manuellement ce même libellé exact lors d'une revue précédente, on réutilise sa décision (confiance 100 %). Voir §8.

② Aliases manuels (codés)

Une table de ~120 alias couvre les noms courts/abréviations rencontrés régulièrement :

'ipc'           → 'Marseille'    (Institut Paoli Calmettes)
'becquerel'     → 'Rouen'        (Centre Henri Becquerel)
'hegp'          → 'APHP_Pompidou'
'pellegrin'     → 'CHU_Bordeaux' (Hôpital Pellegrin)
'brabois'       → 'CHRU_Nancy'   (Hôpital Brabois)
'haut leveque'  → 'CHU_Bordeaux'

Si l'input contient l'alias en sous-chaîne (après normalisation accents/casse), match avec confiance 95 %.

③ Fuzzy match pondéré (IDF)

Si aucun alias ne matche, on calcule un score 0–1 entre l'input et chaque entrée de la nomenclature. Les tokens rares (becquerel, lacassagne) pèsent plus que les communs (centre, hopital) via une formule IDF :

poids(token) = log(1 + N_total / fréquence(token))

Bonus +0.35 si la ville matche, +0.4 si le code de nomenclature apparaît tel quel dans l'input. Match retenu si score ≥ 0.4.

④ Pas de match → erreur

Si toutes les stratégies échouent, la ligne est marquée erreur, la cellule CLCC reste vide et l'utilisateur doit corriger en revue.

Niveaux de confiance utilisés

ConfianceOrigineStatut
1.00Alias utilisateur (correction passée) ok
0.95Alias manuel codé ok
0.85Alias mappé vers code absent de la nomenclature ok
0.70 – 1.00Fuzzy haute confiance ok
0.40 – 0.70Fuzzy faible confiance warning
< 0.40Aucun match error

5. Agrégation des consommables

La BDD réelle traite les consommables différemment des équipements :

Équipements (lots 1-12, 41 pour BioMol)

1 ligne BDD = 1 BC.
Lot label spécifique (Lot_5), montant TTC réel, quantité effective, désignation détaillée.

Consommables (lots 13-40 pour BioMol)

1 ligne BDD = N BCs agrégés par tuple (CLCC × Fournisseur × Année).
Lot label groupé (Lots_13-40), CATTC = somme des BCs, quantité = 1, désignation = nom du fournisseur.

Cette convention vient de l'observation directe de la BDD ground truth : 174 BCs Promega consommables 2024 sont représentés par 7 lignes BDD (une par CLCC). C'est ce que reproduit l'algo.

Pour Anapath PPE028, la BDD ne fait pas d'agrégation : chaque BC reste une ligne individuelle. La config marché contrôle ce comportement via consommablesGroupedLotLabel et consommablesLotRange.

6. Détection des doublons

Trois niveaux de dédup pour éviter les doublons à l'export :

Niveau 1 — Filtre des fichiers de test/copie

Les fichiers nommés Test-*, Copie de *, * (1).xlsx, Copy of * sont automatiquement filtrés. Si plusieurs reportings du même fournisseur pour la même année existent, on garde le moins « junk ».

Niveau 2 — Dédup intra-fichier (lignes copiées-collées)

Si une ligne du reporting est strictement identique à une autre (même établissement + désignation + référence + qté + montant + date + lot), une seule occurrence est conservée. Sécurise contre les sections collées deux fois.

Niveau 3 — Dédup cross-fichier

Si deux reportings (ex : bilan mid-year + bilan end-year d'un même fournisseur) contiennent les mêmes BCs, le BC n'est compté qu'une fois. Même clé stricte que le niveau 2.

7. Apprentissage automatique depuis la BDD

Au chargement du Suivi_Invest, l'algo scanne les lignes BDD existantes pour apprendre les conventions par marché :

Convention appriseSource
Libellé Marché (col C)Valeurs distinctes col C
Référence PPE/ALPattern PPE\d+ ou AL\d+ dans col E
Type d'équipement par lotVote majoritaire col D pour chaque lot col E
Lot consommables groupéPattern Lots_X-Y dans col E
Taux Gain/Achats par défautValeur Y la plus fréquente
Fournisseurs canoniques vusValeurs distinctes col F
Médiane TTC + écart-type par fournisseurStats sur col L par fournisseur

Cela permet à l'algo de fonctionner pour des marchés sans configuration manuelle (Imagerie, Médecine nucléaire, Radiothérapie, etc.) en s'inspirant des conventions historiques de la BDD.

Priorité de résolution : config manuelle > config apprise > config par défaut dérivée de mockData.js.

8. Mémorisation des corrections utilisateur

À l'étape 4 (revue), si l'utilisateur édite manuellement la cellule CLCC d'une ligne, l'algo enregistre la correspondance (libellé source brut → code corrigé) dans le localStorage du navigateur.

Au prochain import, la même chaîne sera reconnue automatiquement avec confiance 100 %. L'algo s'enrichit donc tout seul avec l'usage.

Exemple : vous corrigez « GH HEGP-BROUSSAIS » → APHP_Pompidou. Au prochain reporting MM France, ce libellé sera trouvé automatiquement, sans intervention.

9. Code couleur des cellules à la revue

À l'étape 4, chaque ligne candidate peut être dépliée pour voir les 28 colonnes BDD. Chaque cellule porte un statut visuel :

CouleurStatutSens
vertok Valeur présente avec confiance ≥ 70 %
orangewarning Valeur présente mais confiance 40–70 % : à vérifier
rougeerror Valeur attendue mais vide, ou confiance < 40 % : bloquant
blancempty-ok Cellule optionnelle laissée vide volontairement
grisformula Calculée par Excel à l'ouverture du fichier

L'utilisateur peut éditer toutes les cellules non-formule avec un éditeur adapté (text, number, date picker, dropdown CLCC). Les modifications sont persistées dans la file d'attente d'export.

10. Export Excel et préservation des styles

L'export utilise la bibliothèque xlsx-js-style (extension de SheetJS) qui :

  • Lit le Suivi_Invest source en conservant cellStyles: true
  • Hérite du style de la dernière ligne data pour chaque nouvelle ligne ajoutée (couleurs, polices, alignements, bordures du template UNICANCER conservés)
  • Écrit en re-sérialisant les styles (le xlsx standard ne sait pas le faire, d'où l'utilisation de la version js-style)

Les formules des colonnes calculées (N, Q, R, S, T, U, V, W, X, Z, AA, AB) sont écrites comme expressions canoniques (=L1893+J1893, etc.) avec une valeur cachée 0. Excel recalcule à l'ouverture.

Rapport qualité post-export

Une modale s'ouvre automatiquement après chaque export et affiche :

  • Origine des lignes : 100 % auto / auto-éditées / saisies manuelles
  • Statut initial : combien étaient OK, en avertissement, en erreur (corrigées avant export)
  • Confiance moyenne du matching CLCC sur l'ensemble
  • Répartition par marché si plusieurs marchés exportés ensemble
